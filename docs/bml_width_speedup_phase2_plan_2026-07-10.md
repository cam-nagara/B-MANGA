## 1. 背景

フェーズ1で線幅の頂点単位計算はnumpy化されるが、残る主要コストは次の2つと推定される（推定であり、だからこそ先に実測する）:

1. **頂点グループへの書き込み**: `vertex_analysis._write_vertex_group_weights` は `vg.add([i], w, "REPLACE")` を1頂点ずつ呼ぶPythonループ。頂点グループには `foreach_set` が存在しないため、numpy化できず残っている。
2. **反映一式の周辺コスト**: スタイルウェイト計算、モディファイア更新に伴う再評価など。

関連するユーザーの性能基準（AGENT_INBOX 仕様シグナル 2026-07-07）: tokyo0004級の大規模シーンで「Pencil+4は十秒程度」— B-MANGA Linerも同規模で十秒台の体感が目標。

## 2. ステップA: 実測（ベンチマーク）

### 2.1 やること

`_verify/2026-07-10_bml_width_perf/` にベンチスクリプトを新設し、ヘッドレスBlenderで実測する。

```bash
'/c/Program Files/Blender Foundation/Blender 5.1/blender.exe' --factory-startup --background --python _verify/2026-07-10_bml_width_perf/bench_width_refresh.py
```

- 対象メッシュ: 細分化平面またはグリッドで **1万 / 10万 / 50万 / 100万頂点** の4段階。カメラから2m〜60mに広がる配置（遠近差があるシーンで測る）。加えて「巨大1体＋通常サイズ複数」の混在シーンを1つ。
- 測定対象: 頂点単位モード＋遠近減衰オンで `camera_comp.refresh_objects`（または「反映」経由）の所要時間。
- **内訳を分けて計測**（`time.perf_counter` で区間を仕込む。計測コードはベンチスクリプト側に置き、本体コードへ恒久的な計測は入れない）:
  1. numpy幅計算（width_math）
  2. スタイルウェイト計算（vertex_analysis）
  3. 頂点グループ書き込み（`_write_vertex_group_weights`）
  4. その他（合計との差分）
- 結果は同フォルダに `results.md`（表）と `results.json` で保存。フェーズ1テストの10万頂点スモーク値と整合するか確認。

## 3. ステップB: 汎用float属性 fast-path（GN系ライン種の書き込み高速化）

### 3.1 根拠（調査で確認済みの事実）

- 稜谷線・交差線・選択線のGNツリーは、線幅ウェイトを**モディファイア入力バインディングではなく、ツリー内の `GeometryNodeInputNamedAttribute` ノードで名前文字列により**読んでいる（`inner_lines.py` の `_get_or_create_tree` 内、`intersection_lines.py` のツリー構築部）。名前が一致すれば**本物の頂点グループでなくても、汎用のfloat POINT属性で動く**。
- 実例が既にある: `inner_line_cache.py` は `mesh.attributes.new(_WIDTH_ATTR, "FLOAT", "POINT")` で作った汎用属性を、同じ Named Attribute パターンでGNから読んでいる（`BML_InnerCachedWidth`）。
- 汎用属性には `foreach_set` があり、numpy配列を一括書き込みできる（頂点グループのN回 `vg.add` ループが不要になる）。

### 3.2 やること

1. 稜谷線・交差線・選択線の線幅ウェイト格納先を、実頂点グループ（`VG_INNER_LINE_WIDTH` / `VG_INTERSECTION_LINE_WIDTH` / `VG_SELECTION_LINE_WIDTH`）から**同名の汎用float POINT属性**へ移行する。
   - 書き込み側: `_write_vertex_group_weights` にターゲット別分岐を足すのではなく、「実VGが必要なターゲット（outline/Solidify）」と「属性でよいターゲット」を呼び出し側で分ける構成にする（同型コードの重複を作らない）。属性書き込みは `attr.data.foreach_set("value", numpy配列)` で1回。
   - **アウトライン（Solidify `BML_Outline`）は移行しない**。Solidifyの `vertex_group` プロパティは実頂点グループ必須。
2. **全参照の棚卸しが必須**: `VG_INNER_LINE_WIDTH` / `VG_INTERSECTION_LINE_WIDTH` / `VG_SELECTION_LINE_WIDTH` をリポジトリ全体でgrepし、読み手を全て属性読みへ揃える。少なくとも以下は確認済みの読み手:
   - `inner_line_cache._source_width`: `group.weight(vertex_index)` で実VGを読んでキャッシュへ焼き込む → `mesh.attributes` 読みへ変更。
   - 各ライン種の「VGを保証する」関数（`intersection_lines._ensure_intersection_width_group`、`inner_lines` 内の同種処理）→ 属性の保証へ変更。
   - GNツリー側の Named Attribute ノードは名前一致のため**変更不要**（存在しない場合に1.0へフォールバックする `GeometryNodeSwitch` の挙動も現状維持）。
   - テストコード内でVGを直接読んでいる箇所（フェーズ1新設テスト含む）。
3. **移行時の掃除**: 反映時に旧頂点グループが残っていれば削除する（属性とVGが二重に残ると、どちらが効いているか分からなくなる）。既存 .blend を開いた場合も反映を押せば自然に移行される、を成立させる。
4. スタイルウェイト計算（`vertex_analysis.compute_weights`、フェーズ1で分離済み）のnumpy化も、実測で支配的ならこのステップで行う。

### 3.3 任意（実測でVG書き込みがなお支配的な場合のみ）: Solidify向け量子化バケット書き込み

アウトラインは実VG必須のためループが残る。もし実測で問題になる場合のみ: weightを1024段階に量子化し、同じ値になる頂点indexをまとめて `vg.add(indices, value, "REPLACE")` を1回で呼ぶ（呼び出し回数 N回 → 最大1024回）。線幅精度への影響は0.1%未満で視覚上無害。実装した場合は量子化の旨をコード内コメントで明記。

### 3.4 テスト

- フェーズ1のテスト（`test/blender_b_manga_line_width_falloff_check.py`）を属性読みに追従させ、全ケースPASS。
- 移行検証を追加: 旧VGが残った既存オブジェクトに反映をかけると属性へ移行し旧VGが消えること。稜谷線キャッシュ経由（`inner_line_cache`）でも線幅が正しく反映されること。
- ステップAのベンチを再実行し、`results.md` へ「B適用後」列を追記。

## 4. ステップC: GN内でのカメラ距離動的計算（ライブ追従）— 設計検証のみ。実装前にユーザー報告必須

### 4.1 何が得られるか

- 線幅計算がGN（C++・マルチスレッド）内で毎回自動評価され、**カメラを動かすだけで線幅が追従**する。Pythonの頂点ループ・書き込みが完全に消える。
- 既知P2「フレーム変更時の線幅再計算がカメラ・解像度変更に脆い」（AGENT_INBOX）の恒久解にもなる。

### 4.2 重大なトレードオフ（実装判断をユーザーに仰ぐ理由）

GNツリーにカメラへの依存（Object Infoノード等）を持たせると、**カメラを動かすたびに対象オブジェクトのモディファイアスタック全体（Subsurf・Solidify・ライン生成GN含む）が再評価される**。深度グラフの再評価単位はオブジェクトであり、軽い「幅計算だけ」を別モディファイアに分離しても再評価は丸ごと走る。tokyo0004級の重いシーンでは、ビューポートでカメラを操作するだけで固まる恐れがある — これは2026-07-07以降ユーザーが最も嫌ってきた挙動そのもの。

緩和策の候補: (a) 「カメラ追従」をオブジェクト単位のオプトインのトグルにする（既定オフ、軽いシーン専用） (b) GNのBakeノードで重い上流を固定する（ベイク管理の複雑さが加わる）。

**したがってステップCは、スパイク（使い捨て検証）で成立性とコストを数値化し、結果をユーザーへ報告して指示を待つ。本計画書の範囲で本実装はしない。**

### 4.3 スパイクの内容（`_verify/2026-07-10_bml_width_gn_spike/`）

1. 小シーン（立方体数個＋細分化平面）で、プロトタイプGNツリーを作る:
   - Object Info（カメラ）→ Blender 5.1 の Matrix ソケット系ノードでカメラ空間へ変換 → PERSP は `-Z`、PANO は `Length` で距離を取得 → `W = target_px係数 × wpp(d) × (d_ref/d)^p` をノードで構成。
   - Pythonから押すのはスカラーのみ: `target_px`、`d_ref`、`p`、カメラ型別のpx換算係数（レンズ・解像度・DPI変更時だけ更新）。**魚眼の角度密度は `camera_comp._panorama_radians_per_pixel` の値をソケットへ渡し、式をGN側へ複製しない。**
2. 検証項目:
   - PERSP / ORTHO / 魚眼の3型で、GN計算の線幅がPython実装（フェーズ1）と一致するか（レンダまたは属性読み出しで比較）。
   - カメラをフレームアニメーションで動かした際、線幅が自動追従するか。
   - **コスト計測**: 10万頂点＋Solidify＋ライン生成GNありのオブジェクトで、カメラ移動1回あたりの再評価時間（depsgraph更新時間）を計測。ビューポート操作の体感に耐えるかの判断材料にする。
3. 成果物: スパイク結果のmd（一致可否・再評価コスト・スクリーンショット）。ユーザーへの報告文には「オプトイン方式なら安全に導入可能か」の推奨を1行で添える。
