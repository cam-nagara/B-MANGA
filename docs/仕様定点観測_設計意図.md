# B-MANGA 仕様定点観測 設計意図

> 本書は、B-MANGA における「仕様定点観測」の扱いそのものを定める補助文書である。
> 個別機能の設計意図や実装仕様そのものは、本書ではなく `docs/B-MANGA_設計意図.md` と `docs/B-MANGA-plan.md` を正本として扱う。

## 1. 前提

- グローバル共通ルール（`~/.codex/AGENTS.md`）には、`仕様定点観測` そのものの専用規定は存在しない
- そのため B-MANGA では、`D:/Develop/Meldex-public/app/docs/spec-observation-rules.md` を参照して運用ルールを継承する
- ただし対象プロダクトは B-MANGA であり、Meldex 固有の機能分類やフォルダ構成はそのまま流用しない

## 2. 文書の役割分担

- `docs/B-MANGA_設計意図.md`
  - ユーザーが望んでいる仕様意図の正本
  - ユーザー発言を優先し、粒度を丸めずに残す
- `docs/B-MANGA-plan.md`
  - 現行仕様、現行コード、未達差分、実装順を整理する計画書
  - ユーザー意図と現行コードの差がある場合は、その差分を明示する
- `docs/仕様定点観測_設計意図.md`
  - 仕様定点観測の運用方針自体を定める補助文書
- `docs/spec-snapshot-*`
  - 必要になった時だけ作る実態スナップショット履歴
- `docs/spec-snapshot-*-audit-*.md`
  - 設計意図と実態スナップショットの照合レポート履歴

## 3. 既定の運用

### 3.1 実行条件

- 仕様定点観測の各作業は、**ユーザーの明示的な指示があった時だけ** 実行する
- 指示が無い限り、定期的な自動更新や一括再生成はしない

### 3.2 分離する3作業

- **設計意図更新**
  - ユーザーが何を望んでいるかを更新する
- **実態スナップショット**
  - その時点のコード実態を記録する
- **照合**
  - 設計意図と実態スナップショットの差を整理する

この 3 作業は混同せず、必要時のみ個別に行う。

### 3.3 差分更新の原則

- 既存文書を必ず先に読む
- `反映済みログ範囲` より後の差分を一次探索対象にする
- 差分で整合が保てる場合は差分更新に留める
- `反映済みログ範囲` が曖昧、または既存文書に取りこぼしが多い場合は全量見直しに切り替える

### 3.4 複数ツール利用時のログ横断

B-MANGA は ClaudeCode デスクトップアプリ、VSCode の ClaudeCode、VSCode の Codex を併用して開発されているため、仕様定点観測では単一ツールの会話だけを見て完了扱いにしない。

確認対象:

- ClaudeCode デスクトップアプリの B-MANGA セッションログ
- VSCode の ClaudeCode から作られた B-MANGA セッションログ
- VSCode の Codex から作られた B-MANGA セッションログ
- 現在の会話スレッド内のユーザー発言

除外対象:

- `AGENTS.md` や環境情報などの自動挿入コンテキスト
- ツール実行結果、バックグラウンド通知、継続セッション要約
- AI が作った要約文だけで、直接のユーザー発言として確認できないもの

ただし、継続セッション要約にしか残っていない可能性がある話題は、同時期の直接ログを探し、照合できた範囲だけを反映する。

## 4. 情報源の優先順位

仕様判断の優先順位は以下とする。

1. 同一論点についての**最新のユーザー発言**
2. それ以前のユーザー発言
3. `docs/B-MANGA_設計意図.md`
4. 現行コード
5. `docs/B-MANGA-plan.md`
6. 補助調査資料

補足:

- ユーザー発言同士が矛盾する場合は、**最終判断を優先**する
- ユーザーが未指定の細部は、現行コードを参考に計画書側で補う
- 現行コードがユーザー意図に反している場合、設計意図側ではユーザー意図を正とし、計画書側では「未達差分」として明記する

## 5. B-MANGA で特に重視する対象

現時点の B-MANGA では、次を仕様定点観測の重点対象とする。

- 用紙設定
- ページ一覧と見開き
- コマ枠作成・枠線編集
- コマ編集モードと紙面編集モードの往復
- コマプレビュー反映
- ページ一覧ファイルでの作画レイヤー階層
- レイヤーリストとアウトライナーの連動
- テキスト、日本語インライン入力、縦書き
- フキダシ、効果線、詳細設定、右クリックメニュー
- ラスター描画、用紙色、コママスク
- コマ用blendファイル
- B-MANGA と B-MANGA Render の機能分離
- B-MANGA Render と元アドオン、`c00.blend` の連動
- PSD などの出力系機能は B-MANGA Render 側の重点対象として扱う
- `.clip` 直書き検討の延期判断

## 6. 更新単位

- 原則は **B-MANGA 全体ではなく、現在会話で差分が出た機能単位** で更新する
- ただし、今回のように複数機能へまたがる仕様変更が連続している場合は、関連範囲をまとめて更新してよい
- 全機能の全量見直しは、ユーザーが明示した時だけ行う

## 7. 履歴の扱い

- 設計意図文書と計画書は上書き更新してよい
- ただし `spec-snapshot-*` と照合レポートは履歴資料として扱い、**後から上書きして過去の判断を消さない**
- 同日再作成が必要な場合は `-rev2`, `-rev3` を付ける

## 8. 今回時点の適用判断

- B-MANGA では、仕様定点観測の「意図の正本」は `docs/B-MANGA_設計意図.md` とする

- 2026-05-05 時点では、グローバル共通ルールに「仕様定点観測」そのものの専用規定は無い。B-MANGA では本書を補助文書、`docs/B-MANGA_設計意図.md` を意図の正本として扱う
