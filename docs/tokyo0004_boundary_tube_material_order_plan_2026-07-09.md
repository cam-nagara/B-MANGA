# 修正計画書: 境界チューブ併用時のアウトライン欠落（素材スロット順の破壊）

- 作成日: 2026-07-09（Fable 5 調査セッション・実機で機構特定と修正案の動作証明まで完了）
- 対象: `addons/b_manga_line/outline_setup.py`（B-MANGA Liner）
- 発端: AGENT_INBOX「[P1] tokyo0004系アセットの残る線欠落・黒パッチ」の (a)
- 推奨実行モデル: Sonnet 5（原因特定済みの実装作業）

## 症状（確定済み）

tokyo0004 で街灯・消火栓・バナー支柱など「境界辺を持つ開いたメッシュ」のアウトラインが
欠落する（輪郭カバレッジ92.8%止まりの残り約7%）。単一素材のオブジェクトは全滅、
複数素材のオブジェクトは一部素材の面だけ欠落し、同一オブジェクト内で
「箱は描けるのにポールは描けない」ように見えていた。

## 原因（実機で数値証明済み・2件の複合）

### 原因1: Join Geometry が素材スロット表を並べ替える

`_get_or_create_sheet_outline_tree()`（シートアウトラインGNツリー生成）の最後の
Join Geometry ノードへの接続で、コードは「①元メッシュ → ②チューブ(Set Material済み)」の
順に `links.new` しているが、**Blenderのマルチ入力ソケットは後から接続したリンクが
先頭（先に評価）になる**ため、実際の結合順は「チューブ→元メッシュ」。
その結果、GN出力メッシュの素材スロット表は:

- オブジェクトのスロット: `[元素材, BML_Outline, BML_SheetRimHidden, Inner, Intersection]`
- GN出力（Solidifyへの入力）: `[BML_Outline, 元素材, BML_SheetRimHidden, Inner, Intersection]`

と先頭が入れ替わる。後段の Solidify は `material_offset=+1 / material_offset_rim=+2` の
インデックス加算で「シェル面=アウトライン素材／リム面=非表示素材」を割り当てる設計のため、
表がずれると **柱のシェル面が BML_SheetRimHidden（非表示素材）に塗られて消える**。
さらにチューブ自身も Solidify されて「元素材（白）の袖」がチューブを覆い、
過去調査の「白面がアウトライン発光面より6mm手前」レイ列挙結果もこれで説明がつく。
Solidify `offset=+1` で全消失した既往の異常も「袖がチューブを全包囲する」で説明可能。

対象は「境界チューブ(BML_SheetOutline)とSolidify(BML_Outline)を併用するメッシュ」のみ。
閉じたメッシュはチューブが生成されず並べ替えも起きないため無事（欠落分布と一致）。

### 原因2: `material_offset_rim` を絶対インデックスとして設定している

`ensure_sheet_outline()` が `solidify_mod.material_offset_rim = hidden_index`
（BML_SheetRimHidden のスロット番号）を設定しているが、Solidify のこのプロパティは
**絶対番号ではなく「元の面の素材番号への加算オフセット」**。元素材が1個
（元面の素材番号が0）のオブジェクトでは偶然一致して動くが、複数素材では
2番目以降の素材の面のリムが `BML_Outline_Inner` 等の**可視ライン素材**に化ける
（tokyo0004のバナー板の縁に出るマゼンタ色の縁取りとして実機確認済み）。

## 実機証拠（2026-07-09 GUIライブセッション / Blender 5.1.2）

- S_JPStreetLight_0_014（街灯・元素材1個・360頂点）で、Solidify入力の評価メッシュの
  スロット表が上記のとおり入れ替わっていることを確認。柱区間の最終評価メッシュに
  「SheetRimHidden のシェル面96枚」「元素材(白)のチューブ袖224枚」が実在。
- 検証A: `material_offset=-1 / rim=+1`（並べ替え後の表に合わせた値）へ一時変更
  → 柱に完全なアウトラインが出現。
- 検証B: Join への接続順をライブで逆転（チューブを先に `links.new`）
  → スロット表がオブジェクト順に復元され、**正規オフセット(+1/+2)のまま柱・台座・
  ブラケットまで完全なアウトラインが出現**。修正案そのものの動作証明済み。
- 影響クラス: tokyo0004 内で両モディファイア併用は88メッシュ。

## 修正内容

### 修正1: Join への接続順を逆転（本命・実証済み）

`_get_or_create_sheet_outline_tree()` の Join Geometry への2本の `links.new` の
実行順を入れ替える（`set_mat` 出力を先に、`group_in.outputs["Geometry"]` を後に接続）。
マルチ入力ソケットの「後接続が先頭」挙動により、結合順が「元メッシュ→チューブ」となり
GN出力の素材表がオブジェクトのスロット順と一致する。
挙動がAPI仕様として保証されていない点は、修正3の回帰テストの
「評価後スロット順 == オブジェクトスロット順」アサートで恒久監視する。
コメントでこの逆転挙動を必ず注記すること。

### 修正2: リム面を素材数に依存せず非表示にする

`material_offset_rim` はオフセット加算である前提で設計し直す。推奨案:
`_ensure_outline_material_slots()` のアウトライン素材パディング（元素材n個に対し
アウトラインスロットをn個並べる既存方式）と同じ要領で、**BML_SheetRimHidden も
n個パディングし、`material_offset_rim = 2n`（=元面番号i + 2n が常にRimHiddenの
コピー帯 [2n, 3n-1] に収まる）とする**。これで:
- 元面のシェル: `i + n` → アウトライン帯 ✓（既存設計のまま）
- 元面のリム: `i + 2n` → RimHidden帯 ✓（複数素材でも隠れる）
- チューブ面（スロットn）のシェル: `n + n = 2n` → RimHidden ✓（白い袖の解消）

`ensure_sheet_outline()` の `hidden_index` 代入箇所と、
`outline_fast_update.py` の高速更新パス（現状 `material_offset_rim = material_offset` を
設定しており、境界チューブ設計の「リムは二重線にしない」と矛盾）も同じ規則へ統一する。
スロット構成を変えるため `_outline_material_slots_are_stable()` の妥当性も点検すること。

### 修正3: 旧ツリーの自動再構築と回帰テスト

- 保存済み .blend の旧GNツリーはリンク順を保持したままなので、ツリー検証マーカー
  （例: `_SHEET_TUBE_ANGLE_SPLIT_LABEL` の V18 → V19 等のビルド番号更新）を上げて
  次回のライン適用/更新時に必ず再構築させる（AGENT_INBOX既定の世代管理の教訓に従う）。
- 回帰テスト（リポジトリ内フィクスチャで生成、外部パス非依存）を新設:
  1. 開いた単一素材シリンダー＋開いた2素材メッシュへライン適用
  2. 評価後メッシュのスロット順がオブジェクトのスロット順と一致すること
  3. 元面由来のシェル面の素材が BML_Outline 系であること
  4. リム面・チューブ袖面の素材が BML_SheetRimHidden であること
- 手動検証: tokyo0004 で `_verify/2026-07-09_tokyo0004_outline_coverage/run_coverage_check.py`
  を再実行し、カバレッジ92.8%→95%以上（サブピクセル細物を除く）を確認。
  バナー板のマゼンタ縁取りが消えることも目視確認。

## リスク・注意

- ツリー再構築マーカー更新を忘れると既存ファイルで直らない（最重要）。
- Solidify の `material_offset` 系は負値・範囲外で最終スロットへクランプされる。
  クランプ依存の設計にはしないこと（パディング方式はクランプ非依存）。
- 検証で開いていたGUIセッション（tokyo0004）には未保存のライン適用と
  ライブ実験（共有GNツリーのリンク順逆転）が残っている。**購入アセットの
  .blend なので保存禁止**。破棄（保存せず閉じる）してから実装後の検証をやり直すこと。
- 完了後、AGENT_INBOX の (a) 項を完了へ移動し、(b) 黒パッチ項の再評価
  （本修正後に残るか）を次の調査で行う。
