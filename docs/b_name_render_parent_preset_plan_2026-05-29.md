# B-Name-Render 親プリセット（プリセット連続実行）機能 計画書

作成日: 2026-05-29
種別: 計画書（調査セッションで作成 / 実装は開発セッションへ引き継ぎ）
対象アドオン: `addons/b_name_render`

## 1. 目的

複数のプリセットを順番にまとめて実行できる「親プリセット」を作れるようにする。

## 2. 現状（調査結果）

- プリセットは `BNameRenderPreset`（`name` + `commands`: コマンドの `CollectionProperty`）で、**コマンドの並び**そのもの。
- 全プリセットは `BNameRenderState.presets`（`CollectionProperty`）に並ぶ。
- **プリセットが別プリセットを呼ぶ仕組みは無い**。
  - コマンド種類 `COMMAND_TYPE_ITEMS` に「プリセット実行」系は無い。
  - `BNameRenderCommand` にプリセット参照フィールドも無い。
- 「キャラ」「背景」などは **`PRESET_CATEGORY_ITEMS` による表示カテゴリ**（`preset_category_of` がプリセット名の接頭辞で ALL/CHARA/BG/LEGACY/OTHER に振り分ける）。連続実行とは無関係の**表示フィルタ**。

→ ユーザーが「親プリセットにあたる」と認識していた「キャラ／背景」は、実体は表示カテゴリであり、連続実行ではない。本機能は新規追加となる。

## 3. 実現方針（案A: プリセット実行コマンド）

最も素直で柔軟なのは、**「プリセット実行」コマンドを1種類追加**し、親プリセットを「子プリセットを呼ぶコマンドを並べたプリセット」として表現する方式。

### 3.1 データ
- `core.py` `COMMAND_TYPE_ITEMS` に追加: `("RUN_PRESET", "プリセット実行", "")`。
- `BNameRenderCommand` に子プリセット指定フィールドを追加（新規 `target_preset_name: StringProperty(name="実行するプリセット")`。既存 `text_value` の流用は意味が紛れるため非推奨）。

### 3.2 実行
- `command_runner.py` の実行ディスパッチに `RUN_PRESET` を追加。
- 指定名のプリセットを `BNameRenderState.presets` から探し、その `commands` を**順番に実行**する。
- **循環参照・多重展開の防止**:
  - 実行中プリセット名のスタックを持ち、既にスタックにある名前を再実行しようとしたらスキップ＋警告（無限ループ防止）。
  - 直接自己参照（親が自分を呼ぶ）も同様に弾く。
  - **【確定】多段ネスト無し**: 親は子を1段だけ呼ぶ。子プリセット内の「プリセット実行」コマンドは実行時に無視（＋警告ログ）。深さ1固定なので循環は原理的に起きず、上記スタック管理は簡素化（実行中フラグ程度）でよい。
- **`STATE_BEGIN` / `STATE_END` は子に任せる【確定】**:
  - 親プリセットは退避/復元コマンドを持たない。各子プリセットが自前の「退避→処理→復元」で自己完結する（現行の単体実行を子ごとに順番に行うだけ）。
  - これにより `_begin_session` の二重退避問題を回避する。

### 3.3 UI
- `command_ui.py`:
  - `draw_command` に `RUN_PRESET` 分岐を追加し、`target_preset_name` を選べるようにする（自由入力 or 既存プリセット名から選ぶ `search` プロパティが望ましい。自分自身は候補から除外）。
  - `command_summary` / 処理説明表示（別計画書 `b_name_render_card_term_and_help_plan_2026-05-29.md` の説明表示）にも `RUN_PRESET` を追加: 「プリセット『{target_preset_name}』を順番に実行します」。
- 親プリセットの編集は、既存のコマンド編集UIでそのまま行える（特別な親編集画面は不要）。

### 3.4 カテゴリとの関係【新設する・確定】
- 親プリセット用の表示カテゴリを **新設する**。`core.py` の `PRESET_CATEGORY_ITEMS` に1項目（例: `("GROUP", "まとめ", "親プリセットのみ表示")`）を追加し、`preset_category_of` を拡張する。
- 判定方法は (a) プリセット名の接頭辞（例: 「まとめ」「ページ」で始まる）か、(b) プリセットに種別フラグ（`BNameRenderPreset` に bool/enum を追加）を持たせるか。(a) は既存の仕組みと一貫するが命名規約に縛られ、(b) は名前自由だがデータ追加が要る。実装時に決定する。

## 4. 代替案（案B: カテゴリ一括実行）

「キャラ」カテゴリの全プリセットを順に実行する、のように**カテゴリ単位で一括実行**するボタンを設ける案。
- 実装は軽いが、実行順序がプリセットの並び順依存になり、任意の組み合わせ・順序を作れない。
- ユーザー要望「親プリセットのようなもの（任意に組める）」には案Aの方が合う想定。

## 5. 確定事項（2026-05-29 ユーザー回答）

1. **案A で確定**: 任意のプリセットを任意の順で並べて呼ぶ。
2. **多段ネスト無し**: 親は子を1段だけ呼ぶ。子プリセット内の「プリセット実行」コマンドは実行時に無視（＋警告）。循環は原理的に発生しない。
3. **退避/復元は子に任せ、親は持たない**（設計判断）: 各子プリセットが自前で退避→処理→復元する。親は「プリセット実行」コマンドを並べるだけ。
4. **親プリセット用カテゴリを新設する**（§3.4 参照）。

## 6. 検証項目（開発セッションで実機）

- 「プリセット実行」コマンドを並べた親プリセットを実行すると、指定した子プリセットが順番に実行される。
- 存在しないプリセット名を指定しても落ちない（警告して継続）。
- 循環参照（A→B→A）で無限ループにならず、警告して止まる。
- 子プリセットの退避/復元が二重にならず、実行後にレンダー状態が元へ戻る。
- 既存の単体プリセット実行は従来どおり動作する。
