# B-Name 高速化余地の調査報告 (2026-06-12, v0.6.279 時点)

## 標準分量での実測 (追記)

ユーザーの標準分量 (55 ページ / 各ページ コマ 5 / 全コマにフキダシ・全フキダシにテキスト) を
合成した計測 (`test/blender_perf_benchmark.py`) の結果 (改善前):

| 計測項目 | 実測 | 備考 |
|---|---|---|
| 全 55 ページの詳細読込 (ページ用 blend 起動相当) | **7.8 秒** | page.json 読込自体は軽く、プロパティ復元コストが支配 |
| 保存 1 回目 (詳細読込済みの状態) | **28.5 秒** | 大半が Outliner ミラー同期 (詳細を持った状態の全要素 ensure) |
| 保存 2 回目 (変更なし) | 1.6 秒 | page.json 55 本の再書込 + ミラー fast path |
| プレビュー PNG 55 ページ生成 | 9.9 秒 (180ms/頁) | キャッシュ有効時は 0.26 秒 |
| ページ出力 1 ページ (220dpi, グラデ込み) | 0.30 秒 | 高 DPI ではグラデ全画素ループが支配的になる |
| フキダシのスライダー 10 目盛 (形状変更) | 0.39 秒 (39ms/目盛) | |
| フキダシのスライダー 10 目盛 (**色のみ**) | **0.45 秒 (45ms/目盛)** | 形状不変でも全メッシュ再構築が走っている証拠 |
| レイヤー一覧の再構築 1 回 | **0.45 秒** | ほぼ全ての操作後に走る = 操作毎の固定税 |
| ページ追加 1 ページあたり | 0.6〜1.4 秒 | 追加毎にレイヤー一覧再構築+保存が走り、ページ数に比例して悪化 |

**結論**: 標準分量で最も効くのは、(1) ページ用 blend が全 55 ページの詳細を持つ構造の解消
(= 採番のカウンター化 + 自ページのみ読込。読込 7.8 秒・レイヤー一覧 0.45 秒・保存の多くが同根)、
(2) 形状不変の変更で全メッシュを作り直さない署名スキップ、(3) 保存の差分書込。
以下の優先度リストはこの実測に基づき推奨順を据え置きで確定する。

常駐処理 / 再構築系 / IO・出力系の 3 方面を並行調査した結果の統合。
**本書は調査のみで、コードは未変更。** 各項目は行番号非依存 (ファイル:関数) で記述。

## 優先度 A: 体感への影響が大きく、低リスクで効く

### A-1. 用紙ガイド 1px タイマーの全データ走査をやめる
- 場所: `utils/paper_guide_object.py` の太さ更新タイマー (`_thickness_timer` → `view_constant_thickness`)
- 現状: tick ごとに `bpy.data.curves` 全件と `bpy.data.objects` 全件を走査し、ガイドの bevel/半径を書き換える。ズーム値が変わっていなくても毎回実行。
- 改善案:
  1. 直近の「1px に相当する太さ (meters_per_pixel)」をキャッシュし、**変化していなければ即 return** (誤差 1% 未満はスキップ)。
  2. ガイド対象オブジェクトを名前プレフィックスで毎回探すのではなく、生成時に登録するレジストリ (名前リスト) を持つ。
- 効果: アイドル時 CPU をほぼゼロに。ズーム操作中の引っかかり軽減。オブジェクト数が多い作品ほど効く。

### A-2. ファイル読込直後の 0.6 秒連続再描画ループを短縮
- 場所: `utils/layer_stack.py:schedule_layer_stack_sync` (load_post から 6 回 × 0.1 秒で同期+再描画)
- 現状: 6 tick すべてでレイヤー一覧をフル再構築し `tag_view3d_redraw`。点滅問題 (v0.6.171-173) の温床だった構造が残る。
- 改善案: `_remember_stack_signature` を使い、**署名が前回 tick と同一なら以降の tick を打ち切る** (実質 1〜2 回で収束)。
- 効果: ページを開いた直後のチラつき/硬直の短縮。

### A-3. ページ切替時のプレビュー PNG「強制再生成」を差分化
- 場所: `utils/page_preview_object.py:sync_page_previews` (現在ページを `force=True` で毎回再生成)
- 現状: 内容が変わっていなくてもページを開閉するたびに PIL で描き直す。
- 改善案: `page.json` の mtime (とコマサムネイルの mtime) を PNG の mtime と比較し、新しい場合のみ再生成。`force` は描画設定変更時のみに限定。
- 効果: ページ⇄一覧の往復が速くなる (コマ数が多いページほど効く)。

### A-4. 出力時の `bpy.data.objects` 全走査を Collection 参照に
- 場所: `io/export_pipeline.py:_gp_layers` (効果線オブジェクト探索)、ほか `for obj in bpy.data.objects` 箇所全般
- 現状: 1 ページ出力ごとに全オブジェクトを走査して `bname_kind=="effect"` を選別。
- 改善案: 効果線はページ Collection 配下に置かれているので、**ページ Collection からの限定走査**に変更 (または kind→objects の索引を保持)。
- 効果: オブジェクト数の多いシーンでの出力・プレビュー再生成の定数コスト削減。

## 優先度 B: スライダー操作の応答性 (再構築系)

### B-1. フキダシの帯メッシュ群に署名キャッシュを追加
- 場所: `utils/balloon_fill_mesh.py` / `balloon_line_mesh.py` (主線・外フチ・内フチ・多重線・しっぽ各 ensure)
- 現状: ウニフラ系 (`balloon_flash_effect_line_mesh`) だけ署名スキップがあり、**塗り/主線/フチ/多重線は毎 sync で shapely+earcut から作り直し**。色だけ変えても全メッシュ再構築が走る。
- 改善案: `balloon_mesh_signature` の仕組みを流用し、各メッシュ種ごとに「自分に影響するパラメータのみ」の署名を持たせ、未変更ならメッシュ再構築をスキップ (マテリアル色だけ更新)。
- 効果: 詳細設定ダイアログのスライダー操作が大幅に軽くなる (1 フキダシ最大 15 オブジェクトの再構築が条件付きに)。**最も費用対効果が高い改善。**

### B-2. 色・タイトルなど「形に影響しない変更」の経路分離
- 場所: `core/balloon.py:_on_balloon_entry_changed` / `_on_balloon_title_changed`
- 現状: 色変更でもカーブ再生成判定〜全 ensure が走る。タイトル変更でレイヤー一覧全再構築 (`sync_layer_stack_after_data_change`)。
- 改善案: 色系プロパティは「マテリアル更新+再描画」だけの軽量コールバックへ分離。タイトルは該当行のラベル更新のみ。
- 効果: カラーピッカーのドラッグが滑らかに。

### B-3. レイヤー一覧 (`utils/layer_stack.py:sync_layer_stack`) の差分更新
- 現状: 変更のたびに `collect_targets` で全ページ×全コマ×全レイヤーを収集し、一覧と突き合わせ。
- 改善案: 変更種別 (追加/削除/改名/並べ替え) ごとに影響範囲を限定。少なくとも「対象ページのみ再収集」へ。
- 効果: 要素数が多い作品でのあらゆる操作後のもたつき軽減。

### B-4. コマ形状メッシュの署名キャッシュ
- 場所: `utils/coma_plane.py` / `coma_border_object.py` / `coma_mask_object.py`
- 現状: 形状変更のたびに earcut から毎回フル再構築 (キャッシュなし)。
- 改善案: B-1 と同様の署名スキップ。コマ枠ドラッグ中の更新は現状でも実用的なので優先度は中。

## 優先度 C: ファイル切替・保存の I/O

### C-1. 保存時の page.json を「変更があったページだけ」書く
- 場所: `utils/handlers.py:save_scene_work_to_disk`
- 現状: Ctrl+S のたびに全ページの page.json を書き出す (100 ページ = 100 ファイル書込)。
- 改善案: 直近に書いた JSON 文字列 (またはハッシュ) をページごとに実行時保持し、一致すれば書込スキップ。
- 効果: 保存の硬直 (大規模作品で 1〜3 秒) を大幅短縮。v0.6.279 の detail_loaded ガードと自然に併存できる。

### C-2. ページ用 blend の「全ページ詳細読込」の遅延化 (要設計判断)
- 場所: `utils/handlers.py:_reload_all_pages_panels` (ROLE_PAGE は全ページ読込)
- 現状: ページを開くたびに全ページの page.json を読む (100 ページで 100 ファイル read)。
- 制約: フキダシ/テキストの **ID 採番が全ページ走査前提** (過去の ID 衝突バグの再発防止)、ページ出力・見開きも他ページを参照。
- 改善案 (段階的):
  1. ID 採番を pages.json 側のメタ情報 (最大採番値) に移し、全ページ詳細への依存を断つ。
  2. その後、ページ用 blend も「自ページ+オンデマンド」に変更 (v0.6.279 で作品/コマ用に整備した `utils/page_detail.py` の仕組みをそのまま流用可能)。
- 効果: ページを開く時間がページ数に比例しなくなる。**効果は大きいが影響範囲が広いので専用バッチで。**

### C-3. 5 秒周期の Outliner 監視 (`utils/outliner_watch.py`)
- 現状: 件数変化時に `mirror_work_to_outliner` フル再実行。depsgraph ハンドラでも全 managed オブジェクトのスナップショット比較。
- 改善案: depsgraph 側は「更新 ID が managed 集合に含まれるときだけ」スナップショット比較へ早期 return。ドラッグ中の毎フレームコストを削減。

## 優先度 D: 出力 (PNG/PSD/PDF) の速度

### D-1. グラデーション塗りの全画素 Python ループを排除
- 場所: `io/export_balloon.py:_fill_source_image`
- 現状: `for y: for x:` の Python 二重ループ + `putdata` (1536px 角で約 236 万画素)。
- 改善案: 1D グラデーション (線形) なので、勾配方向の LUT を 1 本作り、`Image.linear_gradient` + `rotate`/アフィン変換、または NumPy (`numpy.outer`+`Image.fromarray`) で生成。
- 効果: グラデ付きフキダシがあるページの出力が画素数オーダーで高速化。

### D-2. コマサムネイルの読み込みキャッシュ
- 場所: `utils/page_preview_object.py:_draw_coma_thumb` / `io/export_pipeline.py:_render_coma_preview_layer`
- 現状: プレビュー/出力のたびに毎回 `Image.open`。
- 改善案: (パス, mtime) キーの小さな LRU キャッシュ。

### D-3. `image.point(lambda)` の定数化
- 場所: `io/export_balloon.py` / `export_pipeline.py` の alpha スケール処理
- 改善案: lambda ではなく 256 要素のルックアップテーブル (`point(table)`) を渡す (PIL は table 渡しの方が速い)。微益だが安全。
