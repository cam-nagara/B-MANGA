# フキダシ ジオメトリノード最小化計画 (2026-05-27)

## 目的

フキダシのジオメトリノードグループ (`BName_GN_BalloonCurveRender`, 1393 行で構築) が、画像マスク統一後も残骸 (Raycast マスククリップ / 外フチ・内フチ・多重線 CurveToMesh 系統 / 形状ごとの分岐) を抱えたまま肥大化し、移動・サイズ変更・詳細設定変更の重さの原因になっている。

この計画では、次の方針へ収束させる。

- **形状の計算とそのアウトラインのカーブ生成** はすべて B-Name (Python) 側で実施する。
- **塗り面 / 主線 / 外側フチ / 内側フチ / 多重線** のメッシュも、すべて Python (Shapely + earcut) 側で焼き込む。
- **ジオメトリノード** は「すでにある閉じたメッシュ (主線/フチ/多重線) とすでにある塗り面メッシュに、マテリアルを割り当てて Z を整える」だけに縮小する。
- **コマ内マスク** は引き続き画像マスク方式 (マテリアル α) のみで切る。ノード側の Raycast クリップは完全に削除する。
- **塗りぼかし + ディザ** はマテリアル/シェーダー側に残す (軽い処理)。

ユーザー編集中のカーブ追従は、「主線/フチ/多重線」が現状すでに「編集確定 → 再焼き」の挙動なので、それに合わせて塗りも「編集確定 → 再焼き」に統一する (案 X)。

## 背景

- 直近、フキダシ本体の主線・フチ・多重線は Shapely buffer によるバンドメッシュ方式に統一されてきた (v0.5.46 〜 v0.6.128)。
- v0.6.104 で「フキダシ Mesh のコマ内マスクを画像マスク方式に統一」した時、Mesh オブジェクト側のメッシュくり抜き modifier は撤去された。しかしカーブ実体側の Geometry Nodes グラフには Raycast ベースのマスククリップ経路、外側フチ / 内側フチ / 多重線の CurveToMesh 系統が **削除されないまま** 残っている (`mask_object=None` を渡して休眠状態にしているだけ)。
- 形状ごとに `balloon_line_mesh.is_shapely_line_shape` / `is_shapely_multi_line_shape` で「ノード側を停止」する `override` を渡しているため、ノードと Python の二重メンテになっている。

## 到達目標

- ジオメトリノードグループは「カーブをそのまま受け取り、すでに付随しているメッシュにマテリアルを割り当てて Z を整え、塗りぼかしマテリアルがあれば塗りの alpha 属性を作る」だけに縮小する。
- 旧 Raycast マスククリップ / 外フチ / 内フチ / 多重線 / clipped_fill / main_line_fill / role_radius 分離 ノード経路を完全に削除する。
- 全形状 (楕円・矩形・八角形・カスタム・雲・もやもや・トゲ直線・トゲ曲線・自由形状) で、主線・フチ・多重線・塗りを Python メッシュとして同一経路で焼き込む。
- ユーザーが Blender 標準機能でカーブ制御点を編集した場合、「編集確定 → 再焼き」操作が走るまでメッシュは追従しない (主線/フチ/多重線と同じ挙動)。
- B-Name 無効状態でも、保存済みの Curve + Mesh + マテリアルだけでレンダリングが成立する。
- 一つのフキダシは引き続き 1 つの管理単位として扱う (補助メッシュは内部詳細)。

## 非目標

- ノードグループを完全削除して 0 ノードにすることは目標にしない。塗りぼかしの alpha 属性を作る最小限のノードと、マテリアル割当はノード側に残す。
- 効果線は本計画の対象外 (`docs/balloon_curve_source_plan_2026-05-23.md` §4 の効果線方針に従う)。
- 自由形状用の Path/グリースペンシル登録オペレータの新規実装は本計画では行わない (既存計画 Phase 5 のまま残置)。

## 新方式での Geometry Nodes グループ仕様

### 入出力ソケット

```
Input:
  - Geometry              (必須)
  - 塗り素材              NodeSocketMaterial
  - 外側フチ素材          NodeSocketMaterial
  - 内側フチ素材          NodeSocketMaterial
  - 線素材                NodeSocketMaterial
  - 塗り輪郭ぼかし        NodeSocketFloat (0..1)
  - 塗りぼかしをディザ化  NodeSocketBool
Output:
  - Geometry
```

`hide_in_modifier=True` を維持してパネルからは隠す。

### ノードグラフ

```
Geometry input
  ├── Curve to Mesh (Fill Curve) は使わない (カーブを Mesh 変換しない)
  ├── (現状はカーブ実体 + 別 Mesh オブジェクト群、で構成されているため、
  │    ノードグループ内ではカーブをそのまま通す)
  └── Set Material × 4 (役割別)
        ├── 塗り素材を 塗り面 Mesh 要素にのみ割当 (role 属性 == FILL)
        ├── 外側フチ素材を 外側フチ Mesh 要素に割当
        ├── 内側フチ素材を 内側フチ Mesh 要素に割当
        └── 線素材を 主線/多重線 Mesh 要素に割当
  └── (必要なら) Store Named Attribute "bname_fill_blur_alpha"
Geometry output
```

ただしフキダシは **本体 Curve + 別オブジェクトの Mesh 群** で構成しているので、ノードグループは「Curve をそのまま通す + 塗りぼかしの alpha 属性を出す」だけで足りる可能性が高い。

別 Mesh 群側 (主線/フチ/多重線/塗り面) は **モディファイア無し** に変更し、マテリアル割当のみ行う方が単純で速い。これにより本ノードグループは 大幅に縮小される。

最終形は次のいずれか:

- **案 1: ノードグループ完全廃止**  
  すべての描画責務を「Curve 制御点 + 別オブジェクト Mesh 群 + マテリアル」だけで成立させる。
- **案 2: 塗りぼかし alpha 属性のため最小ノードを残す**  
  Mesh 側で alpha 属性を作る場合、Python 焼き込み時に値を直接書ける ((PF))。これを Python 側でやれば案 1 が成立する。

**計画では案 1 を目標**とし、塗りぼかしの alpha 属性も Python 側で Mesh の頂点属性として焼き込む。

## 実装段階 (Phase)

### Phase A: ノードグラフ縮小 (低リスク, 1 ステップ)

現状すでに override で停止している経路 (外フチ / 内フチ / 多重線 / clipped_fill / main_line_fill / Raycast マスククリップ) を、ノードグラフから削除する。

- `balloon_curve_render_nodes.py` から関連ノード生成関数を削除。
- 入力ソケットを削減 (マスク関連 / 外フチ / 内フチ / 多重線関連は撤去)。
- `GROUP_VERSION` をインクリメントして既存ファイルに古いノードグループが残る場合も再生成。
- 既存 Mesh オブジェクト側の挙動は変えない。
- 主線/フチ/多重線で Shapely 帯メッシュを生成していない形状 (= まだ Python 化されていない形状) を一旦調査。もし楕円/矩形でノード経路に依存している場合は Phase B まで待ち、Phase A では現状互換のまま縮小する。

### Phase B: 全形状の主線/フチ/多重線を Python メッシュに統一

- `balloon_line_mesh.is_shapely_line_shape` / `is_shapely_multi_line_shape` を **全形状で True** になるよう拡張。
- 楕円・矩形・八角形・カスタム・自由形状についても主線/フチ/多重線の Shapely バンドメッシュを生成する。
- 既存の Shapely 経路 (雲・もやもや・トゲ) と挙動が一致することを確認。
- 主線/フチ/多重線でノード側にしかなかった分岐 (例: filled_line_enabled) を Python 側で吸収する。

### Phase C: 塗り面 Mesh を Python で焼き込み (案 X)

- 新規 `utils/balloon_fill_mesh.py` を追加 (1000 行制限を考慮し独立ファイル)。
- フキダシの最終的な外形 (主線が削った穴も含む) を Shapely Polygon として確定し、earcut (`mapbox_earcut` などの wheel か、純 Python 実装) で三角分割。
- できた Mesh オブジェクトを `<balloon>__fill` のような命名で本体 Curve に親付け / コレクション登録 / Z 順 / マテリアル割当 する。
- ノードグループ側からは Fill Curve / clipped_fill 経路を削除。
- 塗り輪郭ぼかしの alpha は、Shapely 輪郭からの距離フィールドとして Python 側で計算し、Mesh の頂点属性 `bname_fill_blur_alpha` に書き込む。
- 塗りぼかしをディザ化する処理 (マテリアル側) はそのまま流用。

### Phase D: マスク経路完全撤去とノードグループの最小化

- ノードグループから「マスク使用」「マスク対象」「塗り切り抜き必要」「切り抜き必要」ソケット、Raycast / Object Info / Delete Geometry / Switch (mask 用) をすべて削除。
- ノードグループはこの時点で「Geometry 入力をそのまま通し、Store Named Attribute を 0 本以上追加するだけ」になる可能性が高い。
- もし完全に何もしなくなったら、本体 Curve から geometry-nodes modifier 自体を撤去する (案 1 成立)。

### Phase E: 機能網羅チェックリスト全項目を実機検証

`test/blender_balloon_node_minimization_check.py` を新規作成し、本ドキュメント §「機能網羅チェックリスト」の 102 項目を上から順に検証する。

検証は次の順で行う:

1. ヘッドレス (`--background`) で構造系 (Mesh の有無 / マテリアルスロット / Z 値 / 親子) を確認。
2. レンダー出力 PNG をスクリーンショットで比較し、画素単位で差分を見る (色や半透明)。
3. UI 操作系 (modal operator) は別途 UI 実機テストを追加。

## 軽量化の見通し

- ノードグループのノード数: 現状 数百ノード → 0〜十ノードに削減。
- 設定変更時のノードグループ再構築コスト: ほぼゼロに。
- カーブ評価コスト: Mesh 側はモディファイアなしの素 Mesh なので評価コストもほぼゼロに。
- 編集中追従: 主線/フチ/多重線/塗り、いずれも「編集確定 → 再焼き」に統一。インタラクション中はノード評価が走らない。

## リスクと対策

| リスク | 対策 |
|--------|------|
| 塗りが編集中追従しなくなる ことへの違和感 | 主線/フチ/多重線がすでに同じ挙動。「編集確定 → 再焼き」が一貫することで、むしろ予測しやすくなる |
| 既存 .blend ファイルとの互換性 (古いノードグループ参照) | `GROUP_VERSION` インクリメントで自動再構築。古いノードグループは削除 |
| earcut 依存追加 | まず純 Python の triangulation (Shapely + mapbox_earcut wheel) で試し、wheel 同梱なら `wheels/` 配下に追加。または Shapely の `triangulate` で代替 |
| 塗りぼかしの距離フィールド計算コスト | カーブ更新時のみ計算され、編集中は走らない。コマ枚数が多くてもページ単位 |
| Phase B/C 移行中の機能損失 | Phase ごとに既存テスト全 Pass を維持。差分は CHANGELOG に明記 |

## ロールバック方針

各 Phase 完了時に commit + tag をつけ、問題発覚時は phase 単位で revert できるようにする。Phase A → B → C → D は順序依存。E は検証なので revert 不要。
