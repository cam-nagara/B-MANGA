# フキダシ線帯の「内側フチ方式」統一 修正計画書 (2026-07-23)

## 0. 一言でいうと

フキダシの 外側フチ・主線・多重線・内側フチ のすべてを、現状の**内側フチと同じ描画方式**
（基準輪郭からの内向きオフセット帯）に統一する。見かけの形状・サイズが変わらないよう、
基準輪郭を裏で線幅ぶん拡大してから帯を切り出す。これにより「角を尖らせる」ON時の
山頂の先端が、内側フチと同じ「**等幅のまま鋭角に折り返す**」形状になる。

- 発端: 2026-07-23 ユーザー指示（確定仕様）。「内側フチの先端形状が理想。内側フチと同じ
  アルゴリズムで線を描き、形状自体を裏で拡大して見かけを保てばよい」と、アーキテクチャ
  まで含めて明示された。
- 対象実データ: test071 p0002 balloon_0004（thorn-curve・尖角ON・主線1mm・内側フチ1mm）。

## 1. 現状の事実（実機実測で裏取り済み）

対象: `balloon_line_mesh_balloon_0004` / `balloon_inner_edge_mesh_balloon_0004`（単位mm、フキダシ中心原点）

- **内側フチ（理想形）**: 外周＝本体輪郭そのもの（山頂 (7.822, 37.642) で鋭角に自然折り返し）、
  内周＝−1mm オフセット（(7.19, 35.43) で鋭角に自然折り返し）。等幅シェブロン。曲線キャップ非適用。
- **主線（問題の形）**: 本体輪郭から外側へ +1mm 成長（`body.buffer(+w).difference(body)` 相当、
  内周＝本体そのもの）。山頂ではミター延長で先端 (8.372, 39.542) まで伸び、内周フォールド
  (7.822, 37.642) から先は幅 2.4mm→0mm へ細る針状ウェッジになる。
- さらに **トゲ（曲線）＋尖角ONだけ**、`utils/balloon_thorn_curve_stroke.py` の
  「先端曲線キャップ」（2026-07-15仕様）が主線系の帯に適用され、ウェッジの脚が内向きに
  曲がって「細りながら垂れる」見た目になる。適用ゲートは
  `utils/balloon_line_mesh.py` の `_curve_thorn_peak_polygons()`（`sharp and shape=="thorn-curve"` のみ）。
  内側フチのビルダー（同ファイルの `body_poly.buffer(-edge_width_m, ...)` 経路）はこれを通らない。
- すぼまりの要因は2つ: ①外側整列のため先端の黒がミター延長で内側フチの約2倍に伸びる
  ②曲線キャップが脚を曲げて細らせる（同モジュール冒頭コメントの「線幅は先端に向かって
  自然に細くなる（ユーザー許容済み）」は **2026-07-23 のユーザー指示で撤回**）。

### 幾何学的な原理（なぜ内側フチだけ理想形か）

多角形のオフセットでは、凸角（山頂）は**内向き**オフセットだと2辺が自然に交差して
等幅のまま鋭角に折り返す（join 不要）。**外向き**だと2辺が離れるため mitre/round 等の
join が必要になり、mitre なら幅0へ収束する針、round なら丸くなる。内側フチは両輪郭とも
内向きオフセットなので、山頂が常に理想形になる。

## 2. 新しい構築規則（ユーザー指定アーキテクチャ）

「見かけ形状」= 従来どおりユーザーが編集・認識する輪郭（フィット枠・塗り・テキスト配置の基準）。
各帯は「**基準輪郭から内向きに幅ぶん切り出す**」方式に統一し、基準輪郭は見かけ形状の
mitre オフセットで作る:

| 帯 | 基準輪郭（裏で拡大） | 帯の占有範囲 |
| --- | --- | --- |
| 主線（幅 w） | 見かけ +w/2 | [見かけ −w/2, 見かけ +w/2]（**線幅中心＝見かけ形状**。ユーザー指定） |
| 外側フチ（幅 eo） | 見かけ +w/2 +eo | [見かけ +w/2, 見かけ +w/2 +eo] |
| 内側フチ（幅 ei） | 見かけ −w/2 | [見かけ −w/2 −ei, 見かけ −w/2] |
| 多重線 | 各リングの外周位置を同じ規則で算出 | 主線と同じ整列規則 |

- 基準輪郭の拡大/縮小は mitre join（`_SHARP_MITRE_LIMIT`）で行う。凸角は同じ角度のまま
  先へ延びるだけなので、内向き帯の折り返しは常に「内側フチの先端形状」になる。
- **曲線キャップ（balloon_thorn_curve_stroke）は廃止**。`_curve_thorn_peak_polygons()` の
  呼び出し（`balloon_line_mesh.py` と `balloon_tail_boolean.py`）を撤去し、モジュールの
  確定仕様コメントに撤回日と本計画書パスを記す（履歴として残すか削除かは実装時判断）。
- 「角を尖らせる」OFF（丸結合）の挙動は従来どおり変更しない。切替スイッチの意味は
  「ON＝等幅のまま鋭角折り返し / OFF＝丸折り返し」になる。

### 帯構築の統一則（2026-07-23 全組み合わせチェックで確定・交差方式は不要）

本計画の技術核心は次の1行に単純化できる（初版に書いた「内向き帯∩外向き帯の交差方式」は
不要と判明したため撤回する）:

```
すべての帯 = base.buffer(d2, mitre) − base.buffer(d1, mitre)   （d1 < d2、符号は任意）
```

- 同一基準形状の mitre オフセット対で挟んだ帯は、**凸角（山頂）でも凹角（谷）でも、
  両境界が同角度の鋭角コーナーを持つ平行なペアになるため、等幅のまま鋭角に折り返す**
  （= 内側フチの先端形状）。これは現行コードの `build_offset_band_polygon()`
  （多重線リング用、`buffer(offset±half)` の差）が既に持っている構造そのもの。
- 現状で先端がすぼまって見える要因は、この統一則から外れている2点だけ:
  ① トゲ（曲線）＋尖角ONの主線系にだけ掛かる先端曲線キャップ（境界の片側規則を壊す）
  ② 主線の整列が「内周=本体そのもの (d1=0)・外周=+w」で、折り返しの黒が内側フチの
  2倍の長さに伸びること。→ キャップ撤去＋中心整列（d=±w/2）で全帯が同一規則になる。
- 「角を尖らせる」OFF は同じ式の join を round にするだけ（両境界が丸コーナーの平行対
  → 等幅の丸折り返し）。**整列変更（中心整列）は ON/OFF 共通で適用する**
  （OFF のときだけ線位置が変わる、という不整合を作らない）。
- 万一、極端形状（自己交差カスタム等）で境界対の平行性が崩れる事例が出た場合のみ、
  予備策として初版の交差方式を局所適用する（既定では使わない）。

## 2.5 全形状 × 全線種 × 全オプションの適合チェック（2026-07-23 実施）

コード実査（`balloon_line_mesh.py` / `balloon_multiline_curve.py` / `balloon_thorn_curve_stroke.py` /
`balloon_tail_boolean.py` / `io/export_balloon.py`）に基づく判定。凡例:
**○**=統一則でそのまま成立 / **改**=成立し、現状より見た目が変わる（変化内容を明記）/
**要**=個別対応が必要 / **外**=今回の対象外（帯を持たない等）

### 多重線・二重線（ユーザー質問への回答）

- **多重線: ○（同じ方向性でそのまま対応可能）**。多重線リングは現行でも
  `build_offset_band_polygon()`＝「本体の buffer(offset+半幅) − buffer(offset−半幅)」で、
  統一則と同じオフセット対構造を既に持つ。山・谷とも各リングが等幅シェブロンで折り返し、
  リング同士は平行・等間隔に入れ子になる（同一基準からの mitre オフセットなので幾何学的に保証）。
  必要な変更は2点だけ: ①リング位置の基準を新整列（主線中心=見かけ形状）へシフト
  ②トゲ（曲線）＋尖角ONでリングにも掛かっている先端曲線キャップの撤去。
- **二重線（line_style=double）: ○**。実装は多重線経路（`balloon_multiline_curve.py` ほか）
  なので多重線と同時に対応される。
- 留意: 滑らか形状（雲・モフモフ・トゲ曲線）の多重線は品質のため bezier 外周/内周を
  B-MANGA 側で算出する経路があり（`balloon_multiline_curve.py`）、シェイプリー経路と
  二本立て。実装時は「トゲ曲線＋尖角の山頂」がどちらの経路でも統一則の折り返しと
  数値一致することを受け入れ基準にする（下記テスト）。
- 留意: `thorn_multi_line_cross_enabled`（トゲ多重線の交差オプション）は実装時に
  実機確認（リング生成の入口が同じなら自動追従の見込み）。
- 内側方向の多重線がトゲ幅より深い場合、そのリングでトゲが消えるのは現状どおり
  （shapely オフセットの自然な退化。仕様変更なし）。

| 線種 | 判定 | 内容 |
| --- | --- | --- |
| solid（実線） | ○ | 本計画の主対象 |
| double（二重線） | ○ | 多重線経路で自動対応（上記） |

## 10. フォローアップ実装記録（2026-07-23、v0.6.573）

v0.6.572 の実機確認でユーザーから「まだ先端がすぼまって見える」「小山幅・小山高が効かない」
の2点の報告を受け、同日中に以下を修正した（詳細は CHANGELOG v0.6.573 の項）。

1. **ミター上限の統一（すぼまりの残りの正体）**: 帯の構築は §1 の統一則どおりだったが、
   鋭角の伸びの上限（Shapely mitre_limit）が主線=50・フチ=10・動的幅経路=8〜10と
   バラバラで、上限の小さい帯だけ非常に鋭い山の先端が平らに切り落とされていた。
   `balloon_tail_boolean.SHARP_TIP_MITRE_LIMIT = 50.0` を正典とし全経路を統一。
   動的幅経路の上限超過処理もShapely互換のベベル（2点切断＋三角形充填）へ書き換え。
   ※上限2.0で先端の伸び自体を切り詰める案は、先端が平らな截断になり理想形と
   異なるため試作の上で撤回（描画画像で確認済み）。
2. **小山幅・小山高スライダーの復旧**: 2026-07-15 のコミット ba81fc98 で入った
   「実効小山幅 = max(小山幅, 小山高)」の品質ガードが原因で、小山幅が実質無効・

## 11. 新方式J (頂点距離方式) の実装記録（2026-07-23、v0.6.576）

v0.6.573 でも先端の伸びが残るとの指摘を受け、伸ばす/丸める/切るの三択が

「角を尖らせる」の第二方式として実装した（初期値は従来方式のまま）。

- 仕様: 山・谷の頂点だけをアンカーとし、二等分線方向へ「オフセット×倍率」変位。
  区間曲線は相似変換で追従。角の位置で外側輪郭の角↔内側輪郭の角の距離が
  線幅×倍率になる。既定倍率: 山1.5 / 谷0.5（スライダーで変更可）。
- 実装: `utils/balloon_anchor_band.py`（正典・新規）、`balloon_line_mesh.py`
  （主線・両フチ・多重線の分岐。帯は谷で正当に分割されるため全ピース採用）、
  `balloon_tail_boolean.mitre_band_polygons`（書き出し分岐）、`export_balloon.py`、
  プロパティ/UI（`core/balloon.py`・`panels/detail_drawers/balloon.py`）。
- 制限: J選択時は谷/山線幅%等の動的幅系は無効（倍率スライダーが代替）。
  破線・点線は従来角処理。アンカーの無い形状は自動フォールバック。
- 検証: `test/blender_balloon_anchor_sharp_method_check.py`（新規）合格、
  既存 `blender_balloon_band_center_alignment_check.py` も合格（既定不変）。
  実機で全帯の入れ子収束を描画確認（bJ_live_full/tip.png）。
